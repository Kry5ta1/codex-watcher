

# Agents 项目协作约定

本文件用于约束当前项目内由 Codex、ChatGPT 或其他自动化代理执行的开发、修改、配置、构建、部署交付和排障工作。

所有 Agent 在执行任务前，应优先阅读并遵守本文档。

---

## 补充规则
对于所有任务，在结束时不需要你启动并提供内置的预览，开发调试过程中可以临时启动，但记得最后一定要关闭结束。用户测试时需要自己手动启动，异常的进程占用会给用户造成困扰。

## 1. 核心原则

### 1.1 先理解，再修改

修改代码、配置或文档前，必须先查看相关文件、目录结构、现有实现和已有约定。

不得在未确认现有实现的情况下，凭空重写逻辑、替换方案或调整结构。

### 1.2 最小必要改动

只修改与当前任务直接相关的内容。

未经明确要求，不得顺手进行以下操作：

* 大规模重构；
* 重命名大量文件或目录；
* 替换技术栈；
* 调整项目整体架构；
* 删除无法确认用途的代码、配置或文档。

### 1.3 改动必须可追溯

涉及功能、代码结构、业务逻辑、接口处理、数据流、主要代码入口、代码与数据边界或关键技术决策的变化，应记录到：

```text id="nwg73z"
docs/development-summary.md
```

涉及接口清单、请求体/响应体示例、字段兼容、外部服务差异、上游异常现象、历史联调记录或接口专项排障事实，应记录到：

```text
docs/api-notes.md
```

涉及安装、配置、运行、构建、部署交付、联调或排障方式的变化，应记录到：

```text id="z3ad3r"
README.md
```

### 1.4 文档职责清晰，避免重复

同一条信息只应放在最合适的文档中。

* `README.md` 负责说明：**如何使用、运行、构建、部署交付和排障**。
* `docs/development-summary.md` 负责说明：**项目如何演进的历史、主要代码入口、代码与数据边界、为什么这样实现**。
* `docs/api-notes.md` 负责说明：**接口清单、请求体/响应体边界、字段兼容、外部服务差异和历史联调记录**。
* 专用文档负责说明：**单一功能或单一接入面的完整协议**，例如 `preview.md` 负责 `/preview` 页面型坐标框线预览接入。

如果某条信息同时涉及两个文档，应按用途拆分：

* 操作步骤写入 `README.md`；
* 演进历史、技术原因、主要代码入口、代码与数据边界、代码影响和后续维护注意事项写入 `docs/development-summary.md`。
* 接口请求体、响应字段、字段兼容、上游差异和历史联调事实写入 `docs/api-notes.md`。
* 某个独立功能的完整接入协议、长示例和调用方式写入该功能的专用文档，并在 `README.md` 和 `docs/api-notes.md` 中只保留入口链接或摘要。

### 1.5 不泄露敏感信息

不得将真实账号、密码、token、私钥、生产数据库连接串、内部接口密钥等敏感信息写入代码或文档。

示例配置必须使用占位符，例如：

```env id="44cfvj"
APP_ENV=development
API_BASE_URL=https://your-api.example.com
API_TOKEN=YOUR_API_TOKEN
DB_PASSWORD=YOUR_DB_PASSWORD
```

## 2. 文档分工说明

### 2.1 `README.md` 的定位

`README.md` 是项目的**使用、交付和运行入口文档**。

它的读者通常是：

* 新接手项目的开发者；
* 部署人员；
* 测试人员；
* 运维人员；
* 需要快速运行项目的 Agent。

`README.md` 应回答的问题是：

> 这个项目如何配置、如何启动、如何构建、如何部署交付、如何验证、常见问题如何排查？

---

### 2.2 `docs/development-summary.md` 的定位

`docs/development-summary.md` 是项目的**开发演进、代码与数据边界和维护交接文档**。

它的读者通常是：

* 后续继续开发的开发者；
* 需要理解主要代码入口和代码与数据边界的 Agent；
* 排查业务逻辑问题的人；
* 需要知道某个功能为什么这样实现的人。

`docs/development-summary.md` 应回答的问题是：

> 这个项目如何演进？主要代码入口在哪里？数据从哪里进入、如何转换、流向哪里？为什么这样实现？后续维护时需要注意什么？

### 2.3 代码与数据边界的记录要求

`docs/development-summary.md` 应尤其强化以下两类信息，便于后续 Agent 快速判断改动影响范围。

* **主要代码入口**：记录页面、路由、核心组件、服务/API 封装、状态管理、脚本任务、构建或部署脚本等入口路径及职责。入口新增、迁移、合并或废弃时，应说明影响范围和调用关系变化。
* **数据边界**：记录外部接口、本地存储、缓存、文件解析、地图图层、坐标数据、状态管理、权限会话等数据从哪里进入、经过哪些转换或归一化、最终输出到哪里。涉及字段兼容、空值、异常格式、降级策略和错误提示时，应说明边界处理方式。
* **记录粒度**：不写函数级流水账；优先写清路径、职责、输入输出、边界条件、设计原因和维护注意事项。

### 2.4 `docs/api-notes.md` 的定位

`docs/api-notes.md` 是项目的**接口笔记和联调事实文档**。

它的读者通常是：

* 需要对接本地 API 的前端、外部系统或测试人员；
* 需要核对请求体、响应字段和鉴权边界的开发者；
* 需要排查上游接口、第三方服务、数据库写入差异的 Agent；
* 需要追溯历史联调结论的人。

`docs/api-notes.md` 应回答的问题是：

> 当前有哪些本地 API？请求体和响应边界是什么？上游接口有哪些字段差异和兼容策略？历史联调确认过哪些事实？

### 2.5 专用文档的定位

当某个功能或接入面有大量独立协议、长示例、调用方式和字段说明时，可以保留专用文档，例如当前 `preview.md`。

专用文档应满足以下要求：

* 只服务一个明确功能或接入面；
* `README.md` 只放使用入口或链接；
* `docs/api-notes.md` 可保留摘要和交叉引用；
* `docs/development-summary.md` 只记录该功能的代码入口、数据边界、演进历史和设计原因；
* 不在多个文档中复制同一批长示例。

## 3. `README.md` 记录范围

### 3.1 应记录内容

以下内容应记录到 `README.md`：

* 项目简介；
* 技术栈概览；
* 运行环境要求；
* 依赖安装方式；
* 环境变量配置说明；
* 本地开发启动方式；
* 构建方式；
* 测试或检查命令；
* 生产运行方式；
* 部署交付流程；
* 部署前检查项；
* 部署后验证流程；
* 外部服务依赖；
* 接口入口索引、服务地址或代理配置说明；
* 跨域、网络访问、权限访问等联调说明；
* 常见问题排障；
* 日志位置或排查入口；
* 数据库、缓存、对象存储、地图服务、消息队列等外部依赖的连接说明。
* 详细接口文档或专用文档的链接。

### 3.2 不应记录内容

以下内容不应放入 `README.md`：

* 详细代码演进历史；
* 主要代码入口、调用关系和数据边界的长期维护说明；
* 完整接口请求体、响应体和字段兼容长示例；
* 历史接口联调记录和上游差异流水；
* 某个函数、组件或模块的具体实现细节；
* 每次功能修改的流水账；
* 与运行、构建、部署无关的技术讨论；
* 只影响内部实现、不影响使用方式的代码说明。

---

## 4. `docs/development-summary.md` 记录范围

### 4.1 应记录内容

以下内容应记录到 `docs/development-summary.md`：

* 新增、删除或调整的功能；
* 项目演进历史中有长期维护价值的变化；
* 关键页面、模块、组件、服务或脚本的职责变化；
* 主要代码入口变化，包括页面入口、路由入口、服务入口、状态入口、脚本入口等；
* 代码与数据边界变化，包括输入来源、输出目标、转换/归一化、持久化、缓存和异常边界；
* 数据流变化；
* 状态管理变化；
* 接口字段处理逻辑变化；
* 鉴权、权限、token、会话处理逻辑变化；
* 错误处理、异常兼容、降级策略变化；
* 坐标解析、地图渲染、图层展示、文件解析等业务逻辑变化；
* 数据库表、字段、查询逻辑、数据同步逻辑变化；
* 重要技术决策及原因；
* 兼容性处理；
* 已知风险；
* 后续开发建议；
* 与代码改动直接相关的验证结果。

### 4.2 不应记录内容

以下内容不应放入 `docs/development-summary.md`：

* 常规安装步骤；
* 常规启动命令；
* 常规构建命令；
* 常规部署命令；
* 单纯的服务器、网络、防火墙、代理配置；
* 只和机器环境有关、与代码逻辑、数据边界或主要入口无关的问题；
* 完整接口请求体、响应体示例和长调用示例；
* 没有改变代码逻辑的接口联调流水账；
* 没有长期维护价值的临时排障过程；
* 无业务影响的格式化、注释、空行调整。

## 5. `docs/api-notes.md` 记录范围

### 5.1 应记录内容

以下内容应记录到 `docs/api-notes.md`：

* 本地 API 清单；
* 请求方法、路径、鉴权边界和调用前置条件；
* 请求体、响应体、字段类型和值域示例；
* 接口字段兼容关系，例如旧字段、新字段、别名字段和容错字段；
* 外部服务接口差异、上游异常状态、权限类型、白名单、限流、超时等联调事实；
* 与接口直接相关的数据库表字段、写入字段、字段格式和约束；
* 历史联调结论和可复用排障经验；
* 与专用文档之间的摘要和链接。

### 5.2 不应记录内容

以下内容不应放入 `docs/api-notes.md`：

* 安装、启动、构建、部署交付步骤；
* 完整项目演进历史和技术决策长篇说明；
* 主要代码入口清单；
* 与接口无关的 UI 调整记录；
* 真实账号、密码、token、生产连接串或内部密钥；
* 已有专用文档承载的长篇重复示例。

### 5.3 推荐结构

`docs/api-notes.md` 推荐保持以下结构：

```md
# API Notes

## 文档边界

## 鉴权边界

## 本地 API 清单

### `METHOD /api/path`

## 外部服务边界

### 服务名称

## 历史联调记录

### YYYY-MM-DD 主题
```

当某个接口已经有专用文档时，例如 `/preview` 对应 `preview.md`，`docs/api-notes.md` 只保留摘要和链接。

## 6. 文档更新判定规则

每次完成代码、配置或文档改动后，必须检查是否需要同步更新 `README.md`、`docs/development-summary.md` 和 `docs/api-notes.md`。

### 6.1 判定表

| 改动类型         | 是否更新 `README.md` | 是否更新 `docs/development-summary.md` | 是否更新 `docs/api-notes.md` |
| ------------ | ---------------: | ---------------------------------: | --------------------------: |
| 新增功能         |        视是否影响使用方式 |                                  是 |        视是否新增或调整接口 |
| 删除功能         |        视是否影响使用方式 |                                  是 |        视是否删除或调整接口 |
| 修改业务逻辑       |              通常否 |                                  是 |          视是否影响接口边界 |
| 修改主要代码入口     |        视是否影响使用方式 |                                  是 |                      通常否 |
| 修改代码与数据边界    |       视是否影响配置或联调 |                                  是 |          视是否影响接口字段 |
| 修改接口字段处理     |       视是否影响配置或联调 |                                  是 |                          是 |
| 修改鉴权、权限、登录逻辑 |       视是否影响配置或使用 |                                  是 |          视是否影响鉴权接口 |
| 修改环境变量       |                是 |                          视是否影响代码逻辑 |          视是否影响接口联调 |
| 修改运行命令       |                是 |                                通常否 |                      通常否 |
| 修改构建命令       |                是 |                                通常否 |                      通常否 |
| 修改部署交付流程     |                是 |                                通常否 |                      通常否 |
| 修改外部服务地址配置   |                是 |                          视是否影响代码逻辑 |          视是否影响接口联调 |
| 新增外部服务依赖     |                是 |                                  是 |                          是 |
| 新增重要代码依赖     |       视是否影响安装或运行 |                                  是 |          视是否影响接口联调 |
| 调整目录结构       |        视是否影响使用方式 |                                  是 |          视是否影响文档入口 |
| 修改 UI 样式     |              通常否 |                              视影响程度 |                      通常否 |
| 修改文案         |              通常否 |                                通常否 |                      通常否 |
| 修复 typo      |                否 |                                  否 |                          否 |
| 格式化代码        |                否 |                                  否 |                          否 |
| 临时排障但未改变项目   |        视是否形成排障经验 |                                  否 | 视是否形成接口或上游联调事实 |

### 6.2 可不更新文档的情况

以下情况通常不需要更新文档：

* 修复错别字；
* 调整代码格式；
* 删除无用空行；
* 修改局部样式且不影响功能；
* 修改注释但不改变逻辑；
* 修复非常局部且无长期维护影响的问题。

## 7. `README.md` 推荐结构

`README.md` 应保持面向使用者和交付者，不应写成开发流水账。

推荐结构如下：

```md id="do3k29"
# 项目名称

## 项目简介

## 技术栈

## 运行环境要求

## 安装依赖

## 环境变量配置

## 本地开发

## 检查与测试

## 构建

## 生产运行

## 部署交付说明

## 外部服务依赖

## 联调说明

## 常见问题排障
```

### 7.1 运行环境要求

不要特指某一种技术栈，应根据项目实际情况记录，例如：

```md id="p6tby3"
## 运行环境要求

- 运行时：根据项目实际填写，例如 Node.js、Python、Java、Go、.NET、PHP 等
- 依赖管理器：根据项目实际填写，例如 npm、pnpm、pip、poetry、maven、gradle、go modules 等
- 数据库：根据项目实际填写
- 外部服务：根据项目实际填写
```

### 7.2 环境变量示例

环境变量只写说明和值格式，不写真实敏感值。

```env id="z0j3gf"
APP_ENV=development
API_BASE_URL=https://your-api.example.com
API_TOKEN=YOUR_API_TOKEN
```

### 7.3 命令示例

命令应按项目实际情况填写。

```bash id="c7lcue"
# 安装依赖
<install-command>

# 本地开发
<dev-command>

# 检查
<check-command>

# 构建
<build-command>

# 生产启动
<start-command>
```

## 8. `docs/development-summary.md` 推荐格式

`docs/development-summary.md` 应记录有长期维护价值的开发变化，不应记录无意义流水账。

当项目已经形成稳定结构时，可以在文件前半部分维护长期快照章节，例如“首期范围”“主要代码入口”“数据边界”“技术决策”；在后半部分维护“变更记录”，并按日期倒序追加。长期快照用于快速理解当前项目，变更记录用于追溯历史。

新增变更记录推荐使用以下字段；如果当前文档已经采用 `## 变更记录` + `### YYYY-MM-DD 主题` 的结构，可以继续沿用，但条目内容仍应覆盖主要代码入口和数据边界判断。

```md id="saxmjv"
# Development Summary

## 变更记录

### YYYY-MM-DD 主题

### 修改概要

- 本次改动解决了什么问题。
- 本次改动影响了哪些功能或模块。

### 涉及范围

- `path/to/file`
- `path/to/module`
- `path/to/component`

### 主要代码入口

- `path/to/entry`：说明入口职责、调用方和下游影响范围。
- 如无入口变化，可写：无变化。

### 数据边界

- 输入来源：说明接口、本地存储、文件、缓存、状态或外部服务来源。
- 转换处理：说明字段映射、归一化、空值兼容、异常格式处理或降级策略。
- 输出目标：说明数据最终进入的页面、组件、状态、接口、地图图层、文件或缓存。
- 如无数据边界变化，可写：无变化。

### 关键变化

- 说明功能、代码结构、接口处理、数据流、代码与数据边界或业务逻辑变化。
- 说明重要兼容处理、边界条件或降级策略。

### 技术决策

- 说明为什么采用当前方案。
- 如无特殊决策，可写：无。

### 验证结果

- 已执行：`<check-command>`
- 已验证：`<manual-check-description>`
- 未执行：`<reason>`

### 已知风险与后续事项

- 如无风险，可写：暂无。
```

### 8.1 记录原则

记录应满足以下要求：

* 简洁；
* 准确；
* 可追溯；
* 对后续维护有价值；
* 不编造未执行过的验证结果。

不建议写成：

```md id="g6pstv"
今天修改了很多文件，修复了一些问题。
```

建议写成：

```md id="h1bf35"
调整项目详情页的数据加载逻辑，将接口返回的空数组、空对象和字段缺失情况统一做了兼容处理，避免页面在异常数据下白屏。
```

## 9. Agent 执行流程

### 9.1 修改前

Agent 在修改前应完成以下检查：

1. 阅读用户任务；
2. 查看相关文件；
3. 确认项目类型和技术栈；
4. 确认现有代码风格；
5. 确认相关配置文件；
6. 确认项目内是否已有说明文档、约定文件或外部服务资料；
7. 判断本次是否可能影响：

   * 功能；
   * 业务逻辑；
   * 主要代码入口；
   * 代码与数据边界；
   * 数据流；
   * 接口字段；
   * 接口请求体、响应字段或联调记录；
   * 环境变量；
   * 运行方式；
   * 构建方式；
   * 部署方式；
   * 外部服务依赖。

### 9.2 修改中

Agent 在修改过程中应遵守以下规则：

1. 优先保持现有代码风格；
2. 优先复用已有工具函数、组件、模块和配置；
3. 不随意引入新依赖；
4. 不随意改变目录结构；
5. 不删除无法确认用途的代码；
6. 不把临时调试代码留在正式代码中；
7. 涉及接口数据时，应考虑空值、缺失字段和异常格式；
8. 涉及外部服务时，应考虑网络失败、权限失败和配置缺失；
9. 涉及用户可见功能时，应考虑错误提示和降级体验。

### 9.3 修改后

Agent 在修改后应完成以下检查：

1. 检查是否存在明显语法错误；
2. 检查是否破坏现有功能边界；
3. 根据项目实际情况执行必要验证；
4. 判断是否需要更新 `README.md`；
5. 判断是否需要更新 `docs/development-summary.md`；
6. 判断是否需要更新 `docs/api-notes.md` 或专用文档；
7. 汇总修改内容、主要代码入口、数据边界、接口边界、验证结果和风险。

## 10. 验证要求

Agent 应根据项目实际情况选择合适的验证方式。

常见验证包括但不限于：

* 语法检查；
* 类型检查；
* 单元测试；
* 集成测试；
* 构建检查；
* 启动检查；
* 接口联调；
* 页面访问验证；
* 日志检查；
* 数据库连接验证；
* 外部服务连通性验证。

不得编造验证结果。

如果未执行验证，应说明原因，例如：

```text id="hp0xta"
未执行构建：当前环境缺少项目依赖。
```

或：

```text id="g3yo97"
未执行接口联调：当前环境无法访问后端服务。
```

## 11. 接口与外部服务规则

涉及接口或外部服务时，应遵守以下规则：

1. 优先查看项目内已有说明、配置文件、示例文件和历史实现；
2. 不假设接口字段一定存在；
3. 不假设接口一定返回成功；
4. 不把真实密钥、token 或账号写入文档；
5. 影响运行或部署的 base URL、代理配置、鉴权配置、网络访问方式变化，应记录到 `README.md`；
6. 本地 API 清单、请求体、响应体、字段兼容、上游差异、历史联调事实和接口专项排障，应记录到 `docs/api-notes.md`；
7. 接口字段处理、异常兼容、数据转换逻辑、鉴权数据流或代码边界变化，应记录到 `docs/development-summary.md`；
8. 如果项目内说明与实际返回不一致，应在 `docs/api-notes.md` 记录差异和当前兼容方式；如该差异导致代码处理逻辑变化，还应在 `docs/development-summary.md` 记录设计原因；
9. 如果具体项目另有接口文档、外部服务文档或联调说明，应以该项目内明确指定的资料为准，并避免把专用文档中的长示例复制到 README 或开发总结。

## 12. 依赖管理规则

1. 不得随意新增依赖。
2. 新增依赖前，应优先判断项目中是否已有可复用能力。
3. 新增依赖后，应说明：

   * 依赖名称；
   * 引入原因；
   * 使用位置；
   * 是否影响安装、构建或部署。
4. 删除依赖前，应确认没有其他模块仍在使用。
5. 新增或删除依赖后，应根据影响范围更新：

   * 依赖声明文件；
   * 锁定文件；
   * `README.md`；
   * `docs/development-summary.md`；
   * `docs/api-notes.md`，当依赖影响接口调用、外部服务联调或字段兼容时。

## 13. 禁止行为

Agent 不得执行以下行为：

* 未经要求大规模重构项目；
* 未经确认删除重要文件、目录或配置；
* 擅自替换技术栈；
* 擅自改变接口协议；
* 擅自改变数据库结构；
* 擅自引入重型依赖；
* 将真实账号、密码、token、私钥写入仓库；
* 在 `README.md`、`docs/development-summary.md`、`docs/api-notes.md` 和专用文档中重复记录同一批长内容；
* 把环境问题写成代码问题；
* 把代码实现细节写进 `README.md`；
* 把完整接口请求体、响应体、字段兼容长示例或历史联调记录写进 `README.md`；
* 把没有代码边界变化的接口调试流水写进 `docs/development-summary.md`；
* 把部署命令和环境变量说明写进 `docs/development-summary.md`；
* 忽略构建、类型检查、测试或运行错误并声称已完成；
* 编造未执行过的验证结果；
* 在无法确认事实时使用绝对化描述。

## 14. 每次任务完成后的最低检查

完成任意代码、配置或文档改动后，Agent 必须检查以下问题。

### 14.1 是否需要更新 `docs/development-summary.md`

如果本次改变了以下内容，应更新：

* 功能；
* 主要代码入口；
* 代码与数据边界；
* 模块职责；
* 业务逻辑；
* 数据流；
* 接口字段处理；
* 鉴权逻辑；
* 重要兼容策略；
* 技术决策；
* 已知风险。

如果只是运行命令、环境配置或部署步骤变化，通常不更新该文档。

### 14.2 是否需要更新 `README.md`

如果本次改变了以下内容，应更新：

* 安装方式；
* 运行环境；
* 环境变量；
* 启动方式；
* 检查命令；
* 构建方式；
* 部署方式；
* 外部服务依赖；
* 接口联调方式；
* 排障流程。

如果只是内部代码逻辑变化，且不影响使用、运行、构建或部署，通常不更新该文档。

### 14.3 是否需要更新 `docs/api-notes.md`

如果本次改变了以下内容，应更新：

* 本地 API 清单；
* 请求方法、路径、请求体或响应体；
* 鉴权边界或会话要求；
* 接口字段兼容、别名字段或异常字段处理事实；
* 外部服务上游返回差异、权限差异、白名单、限流、超时等联调结论；
* 接口专项排障经验；
* 与专用接口文档的摘要或链接。

如果只是内部代码结构变化，且不影响接口边界、字段、联调方式或历史排障结论，通常不更新该文档。

## 16. 文档维护建议

随着项目复杂度增加，可按需增加以下文档，但不要一开始就过度拆分：

| 文档                        | 适用场景                    |
| ------------------------- | ----------------------- |
| `docs/architecture.md`    | 项目架构较复杂，需要单独说明模块边界和技术选型 |
| `docs/api-notes.md`       | 接口兼容、字段差异、联调问题较多；当前项目已启用 |
| `docs/deployment.md`      | 部署交付流程较复杂，README 已经放不下 |
| `docs/troubleshooting.md` | 排障内容较多，README 过长        |
| `CHANGELOG.md`            | 项目需要面向版本发布记录变化          |
| `<feature>.md`            | 单一功能或接入面需要完整协议、长示例和调用方式 |

在没有必要拆分前，优先保持：

* `README.md` 负责使用、运行、构建、部署交付和排障；
* `docs/development-summary.md` 负责项目演进历史、主要代码入口、代码与数据边界和设计原因；
* `docs/api-notes.md` 负责接口清单、字段兼容、外部服务差异和历史联调记录；
* 专用文档负责单一功能的完整接入协议，例如 `preview.md`；
* 当前 Agent 协作文件负责执行规则。
